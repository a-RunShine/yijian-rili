## Context

这是一个全新的 macOS 应用项目，目标是为用户提供一个极简的工具来创建复习提醒日程。项目使用 SwiftUI 构建界面，通过 Apple 原生的 EventKit 框架与系统日历交互。

当前状态：
- 项目目录已初始化（空项目，仅有 OpenSpec 配置）
- 技术栈：Swift + SwiftUI + EventKit
- 目标平台：macOS 14.0+

## Goals / Non-Goals

**Goals：**
- 提供一个简洁的 macOS 应用窗口，包含标题输入和日期选择
- 一键创建3个复习提醒全天日程（+3天、+7天、+30天）
- 自动在日程备注中标记复习次数
- 显示创建结果（成功/失败/警告）
- 请求并处理日历访问权限

**Non-Goals：**
- 菜单栏常驻模式（未来可考虑）
- iOS / iPadOS 支持
- 自定义复习间隔（固定为3天、7天、30天）
- 自动提醒/闹钟设置（全天日程本身具有提醒效果）
- 历史记录或日程管理功能
- 多日历选择（仅使用系统默认日历）

## Decisions

### 1. 使用 SwiftUI 而非 AppKit
**决策**：采用 SwiftUI 构建用户界面
**理由**：
- SwiftUI 是现代 Apple 平台的首选 UI 框架，声明式语法更简洁
- 本项目界面简单（输入框+日期选择器+按钮+列表），SwiftUI 完全胜任
- 与 EventKit 的异步 API 配合良好（async/await + @MainActor）
**替代方案**：AppKit - 更底层但代码量大，不适合这种简单界面

### 2. 使用 EventKit 直接写入而非 AppleScript
**决策**：通过 EventKit 框架直接操作日历数据库
**理由**：
- EventKit 是 Apple 官方推荐的原生 API，与 Calendar.app 使用相同底层
- 直接内存访问，无需跨进程通信，性能更好
- 支持现代权限模型（iOS 17+/macOS 14+ 的 write-only 或 full access）
- AppleScript 对 Calendar 的支持有限且脆弱（参见 Michael Tsai 的文章）
**替代方案**：AppleScript - 更慢、更不可靠、权限处理复杂

### 3. 使用 Full Access 访问权限
**决策**：申请 `NSCalendarsFullAccessUsageDescription` 权限
**理由**：
- 应用需要读取现有日程来检测重复（同名同日期），以提供用户警告
- 使用 Full Access 权限触发时机更灵活：可以在用户首次点击"创建"按钮时再请求（而非启动时），符合最小权限原则
- 用户明确看到应用功能后才请求权限，授权率更高
**替代方案**：Write-Only - 无法检测重复，但隐私侵扰更小。考虑到查重功能已承诺给用户，选择 Full Access

### 4. 创建全天日程（All-Day Event）而非定时日程
**决策**：创建全天日程而非带具体时间的日程
**理由**：
- 复习提醒不需要精确时间，全天日程自然出现在日历顶部
- 避免时区和时间设置的复杂性
- 全天日程本身就有提醒效果（Calendar.app 会在当天显示）
**替代方案**：定时日程（如早上9点）- 更具体但增加了不必要的复杂度

### 5. 允许重复创建但给出警告
**决策**：不阻止同名日程的创建，但向用户显示警告
**理由**：
- 用户可能确实需要在同一天复习多个主题
- 阻止创建会中断用户流程
- 警告提示让用户有知情权，但保留最终决定权
**替代方案**：完全阻止重复 - 过于严格，可能误伤正常需求

## Risks / Trade-offs

- **[风险] 用户拒绝日历权限** → **缓解**：在界面上提供清晰说明，权限被拒绝后显示指引，提供重新请求按钮
- **[风险] 系统默认日历不存在或被删除** → **缓解**：检查 `defaultCalendarForNewEvents` 是否为 nil，如不存在则显示错误并引导用户检查日历设置
- **[风险] 日期计算跨越闰年或夏令时** → **缓解**：使用 `Calendar.date(byAdding:value:to:)` 而非手动计算时间间隔，由系统处理边界情况
- **[风险] 创建日程后用户看不到（日历未同步）** → **缓解**：创建完成后显示成功提示，包含日历名称和日期列表
- **[风险] 应用签名/公证问题** → **缓解**：使用 Xcode 标准模板，无需特殊 entitlements（仅日历权限）

## Migration Plan

本项目为全新应用，无需迁移。部署步骤：
1. 使用 Xcode 构建并运行
2. 用户首次点击"创建"按钮时请求日历权限（非启动时）
3. 用户授权后继续创建日程
4. 成功创建后清空标题输入框并将日期重置为今天

## 创建后操作

- **成功时**：清空标题输入框，日期选择器重置为今天，显示成功提示，允许用户继续创建下一个复习计划
- **失败时**：保留用户输入，显示错误信息，允许用户修改后重试
- **部分成功时**：保留未成功项目的输入，显示已创建和未创建的列表，提供重试选项

## 已解决的 Questions

1. **深色模式**：支持。使用 SwiftUI 默认配色方案，无需自定义颜色，自动适配系统浅色/深色模式。
2. **窗口大小**：固定为 400x500 的小窗口，适合输入且不占屏幕空间。不可调整大小。
3. **键盘快捷键**：支持 Cmd+Enter 快速创建。Tab 键在标题和日期间切换。

## 日期计算规则

- 使用 `Calendar.date(byAdding: .day, value: n, to: baseDate)` 计算
- 所有偏移均为"加 N 天"（非加月），避免跨月歧义
- 由 `Calendar` 自动处理闰年、夏令时、跨月等边界情况
- 示例：1月31日 + 30天 = 3月1日（或3月2日闰年）

## 补充风险

- **[风险] 部分失败导致数据不一致** → **缓解**：每个日程独立保存，失败时不回滚已成功创建的日程。用户可手动删除或通过应用提供的"删除已创建"功能回退
- **[风险] 用户误创建** → **缓解**：成功创建后显示确认信息，包含日历名称和具体日期列表，用户可直观核对。不提供撤销功能（需手动在 Calendar.app 删除）
- **[风险] 中文界面无国际化** → **缓解**：所有 UI 文本使用 `NSLocalizedString`，当前仅提供中文，为未来本地化预留接口
